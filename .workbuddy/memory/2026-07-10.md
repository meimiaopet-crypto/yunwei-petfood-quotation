### WorkBuddy IDE safe-delete shim 终止 Next dev
- 现象：dev server 启动后立刻挂掉，日志 `Error: [safe-delete][SAFE_DELETE_BULK_GUARD_ERROR] spawnSync D:\node.exe EPERM`
- 根因：WorkBuddy 通过 `NODE_OPTIONS=--require=...genie-safe-delete.cjs` 注入 shim，**拦截所有 Node 子进程的 fs.unlink**。Next.js dev 启动时会清空 `.next/server/...` 缓存，shim 内部 `tryTrash` 用 spawnSync 调 `D:\node.exe` 报 EPERM 直接 throw
- 修法：启动 next 进程前清空 `NODE_OPTIONS` 和 `BASH_ENV`，让 Node 进程跑在干净环境
  ```bash
  NODE_OPTIONS= BASH_ENV= node node_modules/next/dist/bin/next dev
  ```
- 已把这段修法写进 `start.bat`（Windows 一键启动）

### 关掉 cmd 窗口 → 服务挂掉 → 端口被占
- 现象：用户关掉跑着 next dev 的 cmd 窗口后，再次 `start.bat` 显示 `EPERM .next\trace`，访问 localhost:3000 拒绝连接
- 根因：关掉 cmd **不会**自动杀掉它启动的子进程。Next.js 进程变成僵尸占着 3000 端口但不再响应
- 复现：连开 2 次后 3000/3001/3002 全部被僵尸进程占满
- 修法：手动清僵尸
  ```powershell
  Get-NetTCPConnection -LocalPort 3000,3001,3002 -State Listen |
    ForEach-Object { Stop-Process -Id $_.OwningProcess -Force }
  ```
  然后**用 AI 的 background bash 跑 next dev**（关掉对话窗口也不会杀进程），或保持 cmd 窗口开着
- **教训**：永远不要通过 `start.bat` → 关闭 cmd 的方式停止 dev server，应该 Ctrl+C 优雅退出

### 🎯 PDF 导出 500 错误彻底修复（生产已验证）
- 提交 `25dc43c`，已 push 到 GitHub main，Vercel 自动部署成功
- **生产验证通过**：`https://yunwei-petfood-quotation.vercel.app/api/quotations/YW-QT-20241201-001/pdf`
  5 组合（en/zh/vi quotation + en/zh pi）全部 **200 + application/pdf + %PDF 头 + %%EOF 尾**
- **根因链（3 个叠加问题）**：
  1. **minified React error #31**：Next 15 内置 react-builtin 是 React 19，产出 `react.transitional.element` 符号；项目装的是 React 18，`@react-pdf/reconciler` 按 `React.version` 选了 reconciler-23（只认 legacy `react.element` 符号）→ 元素不被识别 → error #31
     - 修：**升级 react/react-dom/@types 到 19**（package.json，实测 19.2.7），全链路符号统一，@react-pdf 自动选 reconciler-33
  2. **字体网络依赖失效**：`@react-pdf/font` 从 Google Fonts(gstatic) 拉字体，URL 已 404 → `ConnectTimeoutError`
     - 修：改为**本地内嵌** `assets/fonts/NotoSansSC-{Regular,Bold}.woff2`（QuotationPdf.tsx 用 `path.join(process.cwd(),'assets','fonts',...)`）
     - `next.config.mjs` 加 `outputFileTracingIncludes: {'/api/quotations/**': ['./assets/fonts/**']}` 确保字体打进 serverless function
  3. **Vercel 4.5MB 响应体硬限制**：@react-pdf 整体内嵌字体（woff2 解压成 sfnt，6763 GB2312 汉字 regular+bold ≈ 10-13MB），超过 Vercel Functions 缓冲响应上限 → `FUNCTION_RESPONSE_PAYLOAD_TOO_LARGE`（500 变种）
     - 修：**route.ts 改用 `renderToStream` + `Readable.toWeb()` 流式响应**（`Transfer-Encoding: chunked`，无 content-length）→ 绕过 4.5MB 限制。这是官方推荐方案
- **字体子集化**：`scripts/subset_fonts.py`（fonttools+brotli，venv `C:/Users/Administrator/.workbuddy/binaries/python/envs/default`），字符集 ASCII+越南语+CJK标点+货币符号+GB2312 6763 汉字
  - 注意：子集化 woff2 仅 ~830KB，但 @react-pdf 内嵌时解压成 sfnt，PDF 仍 10-13MB。**子集化没能显著减小 PDF**——真正解决 Vercel 限制的是流式响应
- **教训**：
  - @react-pdf/renderer 与宿主 React 版本必须一致（reconciler 按 React.version 运行时选择）
  - Vercel 大响应体（>4.5MB）一律走 streaming，别 buffer
  - 生产域名根路径 307 → `/dashboard` 是**应用自身重定向**，不是 SSO 部署保护，站点其实可直接 curl 验证
- **待办**：提醒用户 revoke 那个用于 push 的 GitHub token（`ghp_...MJgtK`）；如需进一步减小 PDF 体积，可考虑 vi 用 Latin-only 字体、GB2312 降到一级 3755 字
